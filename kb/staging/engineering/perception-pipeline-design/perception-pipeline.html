CoBot Vision Perception Pipeline — Design Document

This document describes the design and implementation of the vision perception pipeline for the Contoso Robotics CoBot Vision Module VM-200 as deployed on the CoBot Pro 500 and CoBot Pro 700 platforms. The pipeline transforms raw camera data into actionable object poses for the motion planning stack in CoBot OS v4.2. For initial camera calibration and hardware setup, refer to the Vision Module Setup Guide in the Support documentation. For customer-facing specifications of the VM-200, see the Vision Module Technical Brief from Contoso Robotics Marketing.

1. Pipeline Architecture Overview

The perception pipeline is structured as a directed acyclic graph (DAG) of processing nodes, each running as an independent thread within the cobot_vision ROS 2 component. The pipeline must complete a full perception cycle — from image capture to published pose — within a 50 ms hard deadline to maintain synchronization with the 1 ms EtherCAT motion control loop (poses are consumed once per trajectory segment, typically every 50–100 ms).

1.1 Pipeline Stages

  1. Capture (3–5 ms): Synchronized acquisition of RGB and depth frames from camera hardware.
  2. Preprocessing (2–4 ms): Undistortion, rectification, color space conversion, and region-of-interest (ROI) cropping.
  3. Detection (15–25 ms): Object detection using DNN inference to produce 2D bounding boxes and class labels.
  4. Tracking (1–2 ms): Multi-object tracking to maintain consistent object IDs across frames and smooth transient detection failures.
  5. Pose Estimation (10–15 ms): 6-DOF pose computation from detected regions using depth data and geometric model fitting.

Total worst-case latency: 47 ms, within the 50 ms budget with 3 ms margin for scheduling jitter.

3. Preprocessing Stage

The preprocessing stage prepares raw sensor data for DNN inference:

  • Lens undistortion: Brown-Conrady radial-tangential model with 5 coefficients (k1, k2, p1, p2, k3). Coefficients are loaded from the calibration file generated during the hand-eye calibration procedure (see Section 7).
  • Stereo rectification: For the D455, rectification aligns RGB and depth image planes using the factory-stored extrinsic rotation and translation. The reprojection error after rectification must be less than 0.3 pixels.
  • Color space conversion: RGB → BGR (OpenCV convention) and optional RGB → HSV for color-based segmentation tasks.
  • ROI cropping: A configurable 3D workspace bounding box (defined in the robot base frame) is projected into camera image coordinates. Only pixels within this ROI are forwarded to the detection stage, reducing DNN inference time by 30–40% on typical setups.
  • Depth filtering: Temporal averaging (3-frame window) and spatial edge-preserving filter (bilateral, σ_spatial=3, σ_range=25 mm) to reduce depth noise from < 2% to < 0.5% RMS at 1 m.

4. Object Detection

Object detection is performed using a quantized YOLOv8m model optimized for the NVIDIA Jetson Orin NX accelerator integrated into the VM-200 module:

  • Model: YOLOv8m (medium), trained on a Contoso Robotics custom dataset of 120,000 annotated images covering 47 object classes (fasteners, PCBs, connectors, packaging, automotive parts).
  • Input resolution: 640 × 640 pixels (letterbox-padded from camera aspect ratio).
  • Quantization: INT8 calibrated via TensorRT 8.6 post-training quantization. mAP@0.5 drops from 87.3% (FP32) to 86.1% (INT8) — less than 1.5% degradation.
  • Inference time: 18 ms typical on Jetson Orin NX (20W power mode), meeting the 25 ms stage budget.
  • NMS threshold: IoU = 0.45, confidence threshold = 0.35 (configurable per class via /cobot_vision/detection_params ROS 2 parameter).

4.1 Custom Model Training

Users can fine-tune the detection model on application-specific objects using the CoBot Vision Training Toolkit (Python package cobot-vision-train). The toolkit provides annotation tools, augmentation pipelines (random crop, color jitter, mosaic, MixUp), and automated TensorRT export. A minimum of 500 annotated images per class is recommended for reliable detection (≥ 80% mAP@0.5).

5. Point Cloud Processing

Depth data from the D455 is converted to an organized point cloud and processed using the Point Cloud Library (PCL 1.13) and Open3D (0.18):

  1. Depth-to-point-cloud conversion: Using the camera intrinsic matrix K, each depth pixel (u, v, d) is back-projected to 3D: P = d · K⁻¹ · [u, v, 1]ᵀ. This runs on the GPU via CUDA at 1.2 ms for a 1280 × 720 depth frame.
  2. Statistical outlier removal: Points with mean distance to 50 nearest neighbors exceeding 2σ are removed. Typical rejection rate: 3–5% of points.
  3. Voxel downsampling: Grid size 2 mm for manipulation tasks, 5 mm for bin-picking. Reduces point count from ~920K to ~150K (2 mm grid) or ~25K (5 mm grid).
  4. Plane segmentation: RANSAC-based dominant plane removal (table/conveyor surface) with inlier threshold 5 mm. Iterative for up to 3 planes.
  5. Euclidean clustering: Remaining points are clustered with a 10 mm distance threshold and minimum cluster size of 100 points. Each cluster corresponds to a candidate object.
  6. Surface normal estimation: Normal vectors computed using PCA over a 15-point neighborhood. Normals are oriented consistently toward the camera viewpoint.

6. Pose Estimation

For each detected object, 6-DOF pose (translation + rotation) is estimated using a hybrid approach:

  • Coarse pose: PointNet++ (MSG variant, 3 set abstraction levels) processes the object point cloud to predict an initial SE(3) pose. Trained on 50,000 synthetic point cloud / pose pairs per object class. Inference time: 8 ms on Jetson Orin NX.
  • Pose refinement: Iterative Closest Point (ICP) alignment between the observed point cloud and the known CAD model, initialized from the PointNet++ coarse pose. Convergence criteria: translation residual < 0.1 mm or 50 iterations maximum. Typical refinement time: 3–5 ms.
  • Symmetry handling: Objects with rotational symmetry (e.g., cylinders, spheres) use a reduced pose representation. Symmetry type (none, Cn, S∞) is specified per object class in the model database.

The final pose is published on the /cobot_vision/object_poses ROS 2 topic as a geometry_msgs/PoseArray stamped with the camera frame timestamp. The CoBot ROS 2 Node Reference Manual in the Engineering documentation provides the full topic and service API.

7. Calibration

The perception pipeline requires three calibration procedures:

7.1 Intrinsic Calibration

Camera intrinsic parameters (focal length, principal point, distortion coefficients) are calibrated using a 9×7 ChArUco board with 30 mm squares and 22 mm ArUco markers. A minimum of 30 images captured at varied distances (0.3–1.5 m) and angles (±30°) are required. Reprojection error must be below 0.5 pixels. The D455 ships pre-calibrated but recalibration is recommended annually or after mechanical shock.

7.2 Hand-Eye Calibration

The transformation from the camera optical frame to the robot tool frame is determined using the Tsai-Lenz method (eye-in-hand configuration). The robot moves to 20+ poses while observing a fixed calibration target. CoBot OS v4.2 automates this process via Settings → Vision → Hand-Eye Calibration. Expected translational accuracy: < 1 mm. Rotational accuracy: < 0.3°.

7.3 Extrinsic Calibration (Multi-Camera)

For setups with multiple VM-200 modules (e.g., overhead + in-hand), extrinsic calibration establishes the relative camera transforms using simultaneous observation of a shared calibration target. The calibration graph is solved using bundle adjustment with Levenberg-Marquardt optimization.
